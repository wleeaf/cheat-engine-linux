# Code-vs-docs audit (2026-07-11)

Verification that the ✅ "implemented/verified" claims in `FEATURE_GAP.md` actually
hold against the current source. Method: three independent subsystem auditors read
the real code (grep + read, no build/run) and reported only what does **not** hold.

## Verdict

The **large majority of ✅ claims hold** and were traced to genuine, wired code with
file:line evidence. **One real discrepancy** was found (the debugger's GUI status was
overstated) and has been corrected in `FEATURE_GAP.md`. A handful of minor caveats and
asymmetries are listed below (none are false feature claims).

## Discrepancy (fixed in FEATURE_GAP.md)

**The debugger "Have (verified working in the GUI)" line overstated three items.**
The backends are real and unit-tested, but three of them are **not wired into the
shipped GUI or Lua** (only into `test/main.cpp`):

| Claimed ✅ (GUI) | Reality |
|---|---|
| Software breakpoint (int3/POKETEXT) | Backend real + unit-tested (`debug/debug_session.cpp`), but `DebugSession` is constructed only in tests. The disassembler's right-click "set breakpoint" only appends a passive record — nothing plants an int3 in the running target. Not live in the GUI. |
| Exception breakpoint | Same: real backend, unit-tested, not reachable from GUI/Lua. |
| Breakpoint list | `BreakpointListWindow` + `BreakpointManager` exist, but `applyToThread`/`removeFromThread` (which arm entries onto the target) have **zero callers** — entries are never armed. |

The one debugger feature genuinely functional in the GUI is **HW-watchpoint
find-what-writes / find-what-accesses** (multithread via `PTRACE_O_TRACECLONE`, with
back-disassembly to the accessing instruction) — fully verified. At audit time the
**interactive step-debugger window was not shipped** (backend existed, no GUI).

> **RESOLVED 2026-07-12:** the interactive step-debugger shipped (Tools → Debugger,
> `gui/debuggerwindow.*`). `DebugSession` was rewritten for **all-stop multi-thread**
> (SEIZE all threads + `PTRACE_O_TRACECLONE` + `__WALL`; freeze-the-world on any
> breakpoint), wiring live software breakpoints + continue/step into the GUI.
> Verified by `test_multithread_software_breakpoint` (backend) and an offscreen
> `gui_debugger_smoke` (the real window). So the three discrepancy rows above are
> now genuinely GUI-functional.

## Minor caveats (not false claims; candidates for small fixes)

1. `module!symbol` resolves for **display** but not as an **input** expression (parser
   has no `!` split). FEATURE_GAP doesn't claim it, so an asymmetry, not a broken claim.
2. `/proc/kallsyms` parser exists and is unit-tested but is **not wired** to any live
   feature (only tests call it). Never claimed.
3. Break&Trace (`debug/tracer.cpp`) is **single-thread** (self-flagged TODO).
4. `.CT` freeze **direction** is stored in a non-CE custom tag, so that one field won't
   round-trip to upstream CE (all other `.CT` fields are CE-compatible).
5. Global hotkeys are X11-only with a focused-window Qt fallback on non-X11.
6. The Vulkan overlay layer lives in `platform/`, not `plugins/` (location only).
