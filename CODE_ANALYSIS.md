## How this analysis was produced

A multi-agent audit: **17 independent subsystem auditors** read the relevant source in full and reported findings; **every `critical`/`high` finding was then handed to an adversarial verifier** whose job was to *refute* it against the actual code; a final synthesizer deduplicated findings by pattern. Results were grounded in build/test reality first:

- **Builds cleanly** — incremental `cmake --build` exits 0, fully up to date (`-std=c++23`, Qt6, Capstone 5.0.1, Keystone 0.9.2, Lua 5.3).
- **`cecore_test` passes** — spawns a child, reads/writes its memory via `process_vm_readv`/ptrace, enumerates regions/modules/threads.
- **`scan_test` requires root** — gated; **not run** here, so privileged live-scan paths are unexercised by automated testing.

Because the build is green and the smoke test passes, "won't compile / would crash the test" classes of false positive were excluded by construction, and the verifier was instructed to refute anything contradicting that.

## Executive summary

cecore is an ambitious, broad, and largely-working Linux-native reimplementation of Cheat Engine: a memory scanner, ptrace debugger, auto-assembler, trainer/table system, Lua scripting, network ceserver client, and a privileged kernel module are all present and mostly functional. Several subsystems are genuinely well-engineered and memory-safe by construction (the scanner's buffer-scan loops, the analysis layer's correct partial-read clamping, and the ceserver/network client's disciplined wire-length validation stand out). The systemic weakness is one sentence: untrusted inputs (.CT/.CETRAINER tables, target-planted ELF/.so files, the network ceserver protocol, and Lua scripts) repeatedly meet privileged root execution, and the habit of trusting input as well-formed recurs across nearly every parser. Two confirmed critical bugs (an attacker-controlled heap overflow in the hand-rolled ELF symbol parser, and a use-after-free of module code on kernel-module unload) plus a confirmed-against-source unvalidated physical-memory ioremap give a malicious target real corruption/escalation surface. Beyond memory safety, the trainer generator splices untrusted CheatEntry value strings into generated C and invokes gcc (arbitrary code execution from a crafted table), and the Lua surface exposes *Local read/write and shellExecute — total host compromise under the untrusted-script-as-root threat model. The codebase is at a mid-maturity stage: solid skeleton and many correct hot paths, but the privileged/untrusted boundaries and the ptrace state machine are not yet production-hardened.

## Architecture assessment

The codebase is cleanly layered: a platform substrate (Linux process/ptrace/injector, kernel-driver ioctl client, ceserver network client) underpins core engines (scanner, debug, autoasm, tables/trainer, symbols, analysis), which are exposed upward through Lua scripting and a Qt6 GUI. The separation is mostly respected — the GUI marshals worker-thread callbacks back to the UI thread correctly, scanner workers write isolated results merged single-threaded after join, and the address-list/MemoryRecord Lua layer re-resolves handles by stable id rather than caching raw pointers. Strengths: defensive wire parsing in the net layer, correct partial-read clamping in analysis, and good overflow guards in stack_trace and scanner inner loops. Recurring design smells: (1) hand-rolled parsers for complex untrusted formats (ELF, XML/JSON CT, GDB-RSP, expression evaluator) that trust on-disk/on-wire fields verbatim; (2) a fragile ptrace state machine duplicated across DebugSession, the injector, linux_process, CodeFinder, and Tracer, each ignoring ptrace returns and wait status; (3) a divergence between validation paths and execution paths (autoasm check() runs preprocessors that execute() does not); and (4) Lua's C ABI boundary lacking exception firewalls and bounds gating. Privilege/capability gating is applied carefully only in the kernel module's CAP_SYS_ADMIN checks; the userspace layers assume root and rarely defend predictable file paths.

## Top risks

1. **Attacker-controlled heap buffer overflow in ELF symbol parser (CONFIRMED critical)** — _symbols/elf_symbols.cpp_  
   numSyms = sh_size/sh_entsize sizes a numSyms*24-byte buffer, but f.read writes sh_size bytes into it; both fields come verbatim from an untrusted on-disk ELF. A target can plant a crafted .so and trigger a controlled heap overflow in a tool that runs as root and loads symbols for arbitrary modules.
2. **Use-after-free of kernel module code on unload via /proc hook (CONFIRMED critical)** — _kernel/cecore_kmod.c_  
   The /proc iterate_shared hook points the live inode at module text without refcount-pinning or synchronizing against in-flight directory iterators; unloading frees the text while iterators may still call it (plus a dentry refcount leak). Ring-0 UAF — kernel crash or exploitation.
3. **Arbitrary physical-memory read/write via unvalidated ioremap** — _kernel/cecore_kmod.c_ (structured finding, severity critical, automated verdict: *unverified*; **confirmed present by orchestrator direct read**)  
   `cecore_access_physical` (`cecore_kmod.c:85-102`) ioremaps a fully caller-controlled physical address with no `pfn_valid`/`page_is_ram`/range check (none exist in the file). Gated only by `CAP_SYS_ADMIN`, but it turns the driver into an arbitrary phys-mem primitive that can corrupt kernel structures or leak secrets.
4. **Arbitrary code execution from a crafted .CT via the trainer generator** — _core/trainer.cpp:184,233_ (structured finding; verifier **downgraded critical→low**)  
   The trainer code generator splices the untrusted CheatEntry value string verbatim into generated C source and invokes gcc — an RCE channel. The verifier downgraded it because it fires only when the user *deliberately compiles a trainer* from the table (not on mere table load) and paths/descriptions are otherwise shell-quoted/C-escaped; it remains a real injection point worth closing.
5. **Lua *Local read/write family and shellExecute = total host compromise (prose-only, no structured finding)** — _scripting/lua_bindings.cpp_  
   Under the stated untrusted-script-as-root threat model these CE-compat bindings give a one-line script full host control (local memory r/w of the cecore process and shell execution); the most direct privilege-escalation surface in the scripting layer.
6. **LBR drain reads perf records out of bounds (CONFIRMED high)** — _debug/lbr_tracer.cpp_  
   The ring-buffer drain uses the wrong bound and ignores ring-wrap, treating each record as contiguous; records straddling the mapped data region produce a genuine OOB read of kernel-shared mmap memory.
7. **Use-after-free of Qt widgets through Lua GUI bindings (CONFIRMED high)** — _scripting/lua_gui.cpp_  
   Widget userdata wraps a raw QWidget* with no destroyed-time invalidation; once a WA_DeleteOnClose form or a parent widget is destroyed, any subsequent script call dereferences freed memory — reachable from ordinary GUI scripting.
8. **autoasm check() and execute() diverge: syntax-OK scripts break or misbehave at injection (CONFIRMED high)** — _core/autoasm.cpp_  
   The {$lua}/{$if}/{$ccode}/anonymous-label preprocessors run only in check(); execute() feeds an unexpanded script to the parser, so a script that passes validation can fail or inject wrong code. Combined with the zero-filled original-byte bug, the disable/restore path can corrupt the target.
9. **Untrusted size/count fields trigger unbounded allocations across many entry points** — _ceserver_client, mono_debugger_client, scanner, lua_streams, cli/main, gui/memoryfill, analysis/managed_runtime_  
   A recurring pattern where a wire/script/argv/region size drives resize/reserve with no upper bound — a malicious server or input yields multi-GB allocation and bad_alloc termination (DoS). Widest pattern in the codebase; individually medium but collectively a top operational risk.
10. **Untrusted Lua executed with full standard library at debugger privilege** — _debug/breakpoint_manager.cpp, scanner/memory_scanner.cpp_  
   Breakpoint conditions sourced from CT files run under luaL_openlibs (os/io/loadfile available), giving arbitrary code execution from a loaded table; the scanner's per-candidate Lua VM compounds this with severe performance cost.

## Cross-cutting themes

These patterns recur across multiple subsystems and are the highest-leverage things to fix once, centrally.

### 🔴 CRITICAL — Untrusted size/count fields drive unbounded or overflowing allocations

**Affected:** symbols/elf_symbols (CONFIRMED CRITICAL heap overflow: sh_size bytes read into a numSyms*24 buffer), platform-net ceserver_client virtualQueryExFull, mono_debugger_client, scripting/lua_streams, cli/main, gui memoryfill, scanner (buf.resize(region.size) per worker; All-types magnitude filter), analysis/managed_runtime, kernel-module integer-overflow address arithmetic

**Recommendation:** Establish a single rule at every trust boundary: validate every length/count/entsize/size field against a sane upper bound AND against the actual available bytes BEFORE sizing any allocation or read. For ELF, require read-size == buffer-size and cap section sizes against file length. Wrap allocation sites that can still throw (bad_alloc/length_error) so a hostile input is at worst a clean error, never termination or corruption.

### 🔴 CRITICAL — Untrusted tables/scripts reach code execution, amplified by routine root operation

**Affected:** core/trainer (splices CheatEntry value verbatim into generated C, invokes gcc — RCE from a crafted .CT), scripting/lua_bindings (*Local read/write and shellExecute = host compromise), debug/breakpoint_manager (runs CT-sourced Lua conditions with full luaL_openlibs stdlib), scanner/memory_scanner (Custom first-scan formula runs a full Lua VM per candidate)

**Recommendation:** Treat CT/trainer files and Lua scripts as untrusted code under the as-root threat model. Never compile/splice table-derived strings into C+gcc; if codegen is required, emit only from a strictly validated grammar. Sandbox Lua condition/formula evaluation (restricted environment without os/io/package/loadfile), gate shellExecute/*Local behind explicit opt-in, and surface a clear trust warning when loading third-party tables.

### 🟠 HIGH — ptrace state-machine fragility: ignored returns, unvalidated wait status, PTRACE_ATTACH races, wrong-thread operations

**Affected:** platform/linux/injector (remoteSyscall/remoteCall ignore every ptrace return and wait status; hard-coded x86_64 ABI with no is64bit guard), platform/linux/linux_process (PTRACE_ATTACH per-call, can restore corrupted opcode), debug/debug_session (WNOHANG step then unconditional GETREGS; eventCb_ under bpMutex_ deadlock), debug/code_finder & debug/tracer (single-thread attach misses other threads; SIGTRAP assumed to be the watchpoint without checking DR6; no exit check after CONT), gui/registereditor (attaches pid but operates on tid)

**Recommendation:** Consolidate the duplicated ptrace logic into one audited helper. Check every ptrace() return and inspect waitpid status (WIFEXITED/WIFSIGNALED/WIFSTOPPED) before reading registers or restoring opcodes; confirm SINGLESTEP/int3/watchpoint stops via RIP/DR6; migrate PTRACE_ATTACH to PTRACE_SEIZE/INTERRUPT and attach all threads; never invoke user callbacks while holding the breakpoint mutex.

### 🟠 HIGH — Discarded return values and mishandled partial reads silently produce wrong or corrupting results

**Affected:** core/autoasm (original-byte save discards read Result and stores zero-filled buffers on failed reads → disable()/restore corrupts the target), scanner/memory_scanner (unchecked ::read/pread can yield wrong scan results on EINTR short reads), gui-aux memorybrowser/widgets (ignore process_vm_readv partial-read counts). NOTE: analysis/* is the counterexample that correctly clamps to bytes actually returned.

**Recommendation:** Make read/write results load-bearing: on any failed or short read, abort the operation rather than proceeding with a partial/zeroed buffer. Critically, never persist 'original bytes' from a read that did not fully succeed — gate the patch entirely on a verified full read so the restore path can never write garbage back into the target.

### 🔴 CRITICAL — Object-lifetime / use-after-free at framework and module boundaries

**Affected:** scripting/lua_gui (CONFIRMED: widget userdata holds an unguarded raw QWidget* surviving WA_DeleteOnClose / parent destruction), kernel-module (CONFIRMED: /proc i_fop hook not refcount-pinned or synchronized vs in-flight iterators on unload → UAF of module text, plus dentry refcount leak)

**Recommendation:** In lua_gui, guard widget pointers with QPointer (or a destroyed-time invalidation map) and null-check before every dereference. In the kernel module, do not leave a function-pointer hook into module text installed on unload without RCU/refcount synchronization; either pin the module while hooked or restore i_fop and synchronize before module text is freed.

### 🟠 HIGH — Root operation against predictable, world-writable files without O_NOFOLLOW/O_EXCL (symlink/TOCTOU)

**Affected:** gui/mainwindow speedhack (/dev/shm/ce_speedhack, mode 0666, no O_NOFOLLOW), plugins/speedhack (hard-coded world-writable /dev/shm region steerable by any local user), core/ct_file saveProtected/loadProtected (predictable .json.tmp alongside output, cleartext)

**Recommendation:** Open all root-side files with O_NOFOLLOW|O_EXCL (or O_CREAT|O_EXCL), use restrictive modes (0600), place control files in a per-user/per-process private directory rather than shared world-writable /dev/shm, and write temp files via mkstemp in a directory the attacker cannot influence.

### 🟡 MEDIUM — Robustness/defensive-posture gaps: C++ exceptions crossing the C/Lua ABI, missing build hardening, and protocol/format completeness

**Affected:** scripting/lua_bindings & lua_engine (C++ exceptions escape lua_CFunction into C-compiled liblua frames = UB; lua_tostring NULL-deref on non-string error objects), debug/gdb_remote (missing mandatory RSP RLE/escape decoding; unbounded packet reads), autoasm (INCLUDE recursion with no cycle/depth guard; naive substring symbol substitution), build-ci (no -Wall/-Werror/FORTIFY/stack-protector/sanitizers on a root tool; deps pinned to mutable git tags)

**Recommendation:** Add a C++ exception firewall (try/catch → lua_error) around every lua_CFunction body and null-guard lua_tostring; implement RSP RLE/escape decoding with bounded reads; add include depth/cycle limits and token-boundary-aware substitution; and enable -Wall -Wextra -Werror, _FORTIFY_SOURCE, stack-protector, and CI sanitizer/fuzz builds, pinning third-party deps by commit hash.

## Confirmed findings (spotlight) — 5

Findings the adversarial verifier could **not** refute by reading the source. Treat these as the priority queue.

### 🔴 CRITICAL · `symbols/elf_symbols.cpp:70-73` — Attacker-controlled heap buffer overflow reading symbol table (read size != buffer size)
*Subsystem: symbols · category: security*

numSyms is computed as symShdr.sh_size / symShdr.sh_entsize (floor division), the syms vector is sized numSyms * sizeof(Elf64_Sym) = numSyms*24 bytes, but f.read(syms.data(), symShdr.sh_size) writes symShdr.sh_size bytes into it. Both sh_size and sh_entsize are 64-bit fields taken verbatim from the untrusted ELF section header with NO validation that sh_entsize == sizeof(Elf64_Sym). An attacker who sets sh_entsize larger than sizeof(Elf64_Sym) makes the destination buffer strictly smaller than the bytes read: e.g. sh_entsize=48 yields numSyms=sh_size/48 and a buffer of sh_size/2 bytes while reading sh_size bytes — a 2x heap overflow. The cleanest exploit variant sets sh_entsize > sh_size so numSyms==0: the vector is empty (0 bytes) yet f.read still writes up to sh_size bytes past a zero-size allocation. Section data offset/size is otherwise only bounded by EOF (read failbit), so the overflowing bytes are fully attacker-controlled file contents. The tool typically runs as root and parses arbitrary target-module ELF files (which an unprivileged target could plant), so this is reachable heap corruption -> potential code execution. Note divisibility-checking alone (sh_size % sh_entsize == 0) is INSUFFICIENT: sh_entsize=48 is 'divisible' and still overflows.

> **Verifier (high confidence):** Verified the cited code at symbols/elf_symbols.cpp:70-73 in the processSymtab lambda. It reads exactly as claimed: `size_t numSyms = symShdr.sh_size / symShdr.sh_entsize;` (floor division), `std::vector<Elf64_Sym> syms(numSyms);` (numSyms*24 bytes — I compiled and confirmed sizeof(Elf64_Sym)==24), then `f.read(reinterpret_cast<char*>(syms.data()), symShdr.sh_size);` which writes sh_size bytes into that buffer.

The asymmetry is real and unguarded. shdrs are read verbatim from the file (line 51) with zero validation of sh_size/sh_entsize/sh_offset. The ONLY guard touching these fields is line 61 (`if (symShdr.sh_entsize == 0) return;`). There is no check that sh_entsize == sizeof(Elf64_Sym), so whenever sh_entsize > 24 (e.g. 48 -> ~2x overflow) or sh_size is not an exact multiple of sh_entsize, the read writes more bytes than the buffer holds. The extreme case sh_entsize > sh_size gives numSyms==0 (empty/zero-size vector) while up to sh_size bytes are still written — a write entirely past a zero-size allocation. The finding's note that divisibility alone is insufficient is correct (sh_entsize=48 divides sh_size yet overflows).

The post-read `if (!f) return;` on line 74 does NOT mitigate: istream::read writes the available bytes into the destination buffer FIRST (up to EOF), then sets failbit; the check runs after the overflowing write has already occurred. EOF only bounds the byte count, and the attacker controls file size/contents, so the overflowing bytes are fully attacker-controlled file data.

Reachability: parseElfSymbols is reached via loadProcess (iterating attached-process module paths, line 23) and loadModule (line 28). The parsed ELF is an arbitrary target-module file an unprivileged target could plant, and the tool commonly runs as root. So this is a remotely-plantable, attacker-controlled-length, attacker-controlled-content heap overflow in a privileged context — a critical memory-safety/security bug. The string-table read (lines 64-66) is correctly sized and is fine; the symbol read is the defective one.

Build/test reality does not contradict: the overflow only triggers on malicious/corrupt section headers, which the passing cecore_test never feeds, and well-formed system libraries have sh_entsize==24 with sh_size a multiple of 24, so normal runs never hit it. No compile/link implication.

The proposed fix (reject sh_entsize != sizeof(Elf64_Sym), read exactly numSyms*sizeof(Elf64_Sym), and bound sh_offset+size by file size) is correct and addresses the issue. Confirmed at the stated severity.

**Fix:** Validate symShdr.sh_entsize == sizeof(Elf64_Sym) (reject otherwise), then read exactly numSyms * sizeof(Elf64_Sym) bytes, never sh_size: e.g. `if (symShdr.sh_entsize != sizeof(Elf64_Sym)) return; size_t numSyms = symShdr.sh_size / sizeof(Elf64_Sym); std::vector<Elf64_Sym> syms(numSyms); f.read(reinterpret_cast<char*>(syms.data()), numSyms * sizeof(Elf64_Sym));` Also reject sections whose sh_offset+computed-size exceeds the actual file size before reading.

### 🔴 CRITICAL · `kernel/cecore_kmod.c:311-328, 421-425` — Use-after-free of module code on unload: /proc i_fop hook is not refcount-pinned or synchronized against in-flight iterators
*Subsystem: kernel-module · category: concurrency*

cecore_install_proc_hook() copies /proc's file_operations into the static struct cecore_proc_fops, overwrites iterate_shared with cecore_hooked_iterate_shared (module text), and points the live /proc inode->i_fop at &cecore_proc_fops. Two fatal lifetime problems: (1) cecore_proc_fops is duplicated from the original via `cecore_proc_fops = *cecore_proc_fops_orig;` (line 312), so its .owner field is the ORIGINAL proc owner, not THIS_MODULE. The module's reference count is therefore never incremented while the hook is installed, so `rmmod cecore` succeeds even while the hook is active and while threads are mid-readdir of /proc. (2) cecore_remove_proc_hook() at module exit (line 321-328) merely restores i_fop with no RCU grace period, no barrier, and no draining of in-flight callers. Any task executing inside cecore_hooked_iterate_shared / cecore_filter_actor (or that already dereferenced i_fop == &cecore_proc_fops and is about to call it) will, after module_exit frees the module's .text/.data, jump to or read freed memory -> kernel use-after-free / arbitrary code execution / panic. Because /proc readdir is extremely hot (ps, top, every shell completion, systemd), the race window is trivially hit. This is reachable by triggering one HIDE_PID ioctl (needs CAP_SYS_ADMIN) then unloading the module while any /proc enumeration runs.

> **Verifier (high confidence):** Verified all cited code in /home/fearth/Projects/cheat-engine/cecore/kernel/cecore_kmod.c.

(a) Code exists as described. Line 312 `cecore_proc_fops = *cecore_proc_fops_orig;` copies the entire original file_operations (including .owner) from /proc's live inode; line 314 overwrites only .iterate_shared with cecore_hooked_iterate_shared (module text); line 315 points the live /proc inode->i_fop at the static &cecore_proc_fops. Lines 321-328 (cecore_remove_proc_hook) merely restore i_fop and NULL the globals with no synchronization. It is called from __exit at line 423 (cecore_kmod_exit), immediately before misc_deregister and module text/data is freed.

(b) Bug is real and reachable. Mechanism (1): the proc-root original i_fop (kernel built-in proc_root_operations) has .owner == NULL, so the copied cecore_proc_fops.owner is NOT THIS_MODULE. I confirmed via grep that `.owner = THIS_MODULE` appears ONLY at line 402 on the /dev misc-device fops, never on cecore_proc_fops. VFS open does fops_get()/try_module_get(fops->owner); with owner==NULL this module is never pinned, so `rmmod cecore` is not blocked while the hook is live. Mechanism (2): grep confirms NO synchronize_rcu, NO try_module_get/module_put, NO barrier, NO refcount around the hook lifetime; the only spinlock (cecore_hide_lock) and the only rcu_read_lock sites (lines 41/43/154/156) protect unrelated paths (the hidden-PID array and the VM/translate ioctls). Any task executing inside cecore_hooked_iterate_shared / cecore_filter_actor, or that captured file->f_op == &cecore_proc_fops and is about to invoke iterate_shared, will read/jump into freed module memory after unload -> kernel use-after-free. Reachable by one HIDE_PID ioctl (CAP_SYS_ADMIN, line 370/376) which installs the hook permanently until exit; /proc readdir is extremely hot (ps/top/shells/systemd), so the unload-vs-iterator race window is trivially hit, and the racing reader need not be privileged.

(c) Severity critical is justified: ring-0 use-after-free reachable on module unload via a high-frequency path, leading to panic and potentially arbitrary kernel code execution. The proposed fix has a minor imprecision (try_module_get is taken at file-open, not per-iterate) but the core diagnosis is correct; the robust remedy is to not hijack a foreign live inode's i_fop at all.

(d) Build/test reality does NOT contradict it: this is a separate out-of-tree Kbuild module (kernel/Makefile, obj-m, produces cecore_kmod.ko) entirely outside the CMake/userspace build. cecore_test/scan_test never load it into the kernel, so the "tests pass" ground truth provides no coverage of this runtime behavior.

**Fix:** Do not hand-patch a foreign inode's i_fop at all; if hiding is kept, pin the module while the hook is live: after duplicating, set cecore_proc_fops.owner = THIS_MODULE so try_module_get/module_put around the fop balances the refcount and rmmod is blocked while iterators run. Even then, restoring i_fop on a shared live inode is racy — prefer not installing the hook lazily/globally. At minimum, protect install/remove with proper synchronization (e.g. an rw semaphore + synchronize_rcu() after restore) and refuse module unload while the hook is installed. The robust design is to drop the global /proc-fops hijack entirely.

### 🟠 HIGH · `core/autoasm.cpp:1315-1349, 1888-1899` — Preprocessors ({$lua}/{$if}/{$ccode}/anonymous labels) run in check() but NOT in execute() — syntax-OK scripts break or misbehave at injection
*Subsystem: autoasm · category: correctness*

expandLuaBlocks(), expandConditionalBlocks(), and resolveAnonymousLabels() are called ONLY inside check() (lines 1894-1898; grep confirms no other call sites). execute() (the real injection path, 1315-1349) runs only preprocessor hooks + expandStructDefinitions, then feeds the script straight to parseLine(). Consequently any script using `{$lua}...{$asm}`, `{$if}/{$else}/{$endif}`, `{$ccode}`, or `@@:`/`@F`/`@B` passes check() (which reports 'Syntax OK') but at execute() time those tokens reach parseLine()'s fallthrough 'everything else is an assembly line' branch (line 851). A `{$lua}` block becomes a bogus asm line handed to Keystone; `@@:` is misread as a literal label and `jmp @F` references an unresolved symbol. Worse, `@@:`/`{$if}` fail *closed* only after Phase-2 allocations (1352-1364) have already run, so the target leaks the cave allocation on the failing path. Every preprocessor unit test (test/main.cpp test_autoasm_lua_blocks, the @@:/@F/@B test, the {$if} test) uses aa.check(), never aa.execute(), so CI never exercises the divergence.

### kernel-module

_Files: kernel-module (cecore_kmod)_

A loadable kernel module exposing /dev/cecore (mode 0600) with ioctls for arbitrary cross-process VM read/write, raw physical-memory read/write, virt->phys translation, and /proc PID hiding. Every dangerous ioctl is correctly gated behind capable(CAP_SYS_ADMIN) against the init user namespace, and the process-VM and translate paths are reasonably careful with task/mm refcounting and copy_*_user return checks. However, two areas are seriously unsafe: the /proc-hook PID-hiding mechanism has a use-after-free on module unload (and a dentry refcount leak), and the physical-memory ioctls perform ioremap on fully attacker-controlled physical addresses with no pfn/RAM validation or overflow checks. Several integer-overflow gaps exist in the address+offset arithmetic.

**Design notes.** Single misc device, one unlocked_ioctl handler dispatching on the magic-encoded cmd. Privilege gating uses capable(CAP_SYS_ADMIN) (init_user_ns, not ns_capable) on every command including PING — good: container root without host CAP_SYS_ADMIN is excluded. Process-VM access uses get_pid_task/put_task_struct + access_process_vm with a bounded 64 KiB kernel bounce buffer and chunked copy_*_user; translate uses get_task_mm/mmput + get_user_pages_remote with proper put_page. The fixed-width __u64/__u32 struct layout makes compat_ioctl = unlocked_ioctl sound. The weak spots are the two "offensive" features bolted on: (1) physical R/W via ioremap of caller-supplied phys addresses with no validation — effectively arbitrary kernel/RAM/MMIO access that bypasses lockdown even for root and can hang hardware; (2) the runtime patching of /proc's inode->i_fop to a module-owned static file_operations whose .owner is the *original* proc owner (not THIS_MODULE), so the module refcount is never pinned while the hook is live — rmmod can free module text out from under in-flight /proc readdir iterations. The hook is installed lazily and only removed at module exit, with no RCU/synchronization against concurrent iterators.

### symbols

_Files: symbols (ELF/DWARF/kernel symbol resolution)_

The symbols subsystem resolves addresses↔names from three sources: ELF symbol tables (.dynsym/.symtab) parsed by hand, DWARF line/function info via libdw, and /proc/kallsyms text. The kernel-symbol and DWARF paths are reasonably solid, but the hand-rolled ELF parser in elf_symbols.cpp consumes fully untrusted on-disk binaries with insufficient bounds validation: it trusts attacker-controlled sh_entsize and sh_size fields, producing an attacker-controlled heap buffer overflow and an out-of-bounds read. Because this tool commonly runs as root and loads symbols for arbitrary target modules (including ones the target process could have planted on disk), these are genuinely exploitable, not theoretical.

**Design notes.** Clean separation: SymbolResolver (raw ELF symbols, address→name + name→address indexes), DwarfInfo/DwarfRegistry (libdw-backed source/line + function names, with a graceful stub build when CECORE_HAVE_DWARF is unset), and KernelSymbolResolver (kallsyms text). All three share the same resolve() shape: std::map<addr,idx> with upper_bound/--it predecessor lookup plus a 0x1000 slack window, and an unordered_map name index that keeps the first binding on duplicates. The DWARF Impl uses unique_ptr+RAII (closeAll in dtor) and proper move semantics — good. The ELF parser's weakness is that it derives in-memory buffer sizes from one field (sh_entsize-derived count) but issues the file read with a different, larger, attacker-controlled field (sh_size); it also constructs std::string directly from a strtab pointer with no enforced NUL terminator. The kernel parser is the cleanest of the three. None of the parsers wrap large allocations in try/catch, and the whole subsystem assumes single-threaded use (no internal locking; callers must not mutate while GUI threads read — the indexes are plain containers).

### autoasm

_Files: autoasm (core/autoasm.cpp, core/autoasm.hpp, core/aa_templates.cpp, core/aa_templates.hpp)_

A hand-written multi-pass CE-style auto-assembler that parses scripts, allocates target memory, resolves labels/defines via a two-pass sizing+emission scheme, runs AOB scans, and injects code (db/dw/dd/dq, NOP, FILLMEM, LOADBINARY, LOADLIBRARY, CREATETHREAD). The directive-parsing and label-resolution machinery is reasonably structured and the syntax-check (check()) path is feature-complete, but the real injection path (execute()) diverges from check() in ways that are both functionally broken and unsafe: the {$lua}/{$if}/{$ccode}/anonymous-label preprocessors run only in check(), so scripts that pass a syntax check can fail or misbehave at injection. The memory-patching paths discard read/write results and save zero-filled "original bytes" on failed reads, so the disable/restore path can corrupt the target rather than restore it. There are also an uninitialized-read UB on malformed alloc() and a sizing-vs-emission base/exception mismatch in READMEM that desyncs label addresses.

**Design notes.** Two-pass design: parseLine() classifies directives and pushes raw asm/marker lines; resolveForwardLabels() walks lines computing instruction sizes to assign label addresses; the Phase-3 emission loop re-walks the same lines actually assembling/writing. Correctness hinges on the sizing pass and emission pass computing byte-for-byte identical sizes for every line — any divergence (READMEM base mismatch, an emission path that no-ops while sizing reserved bytes) silently corrupts every subsequent label address. execute() has no rollback on mid-script failure: the assembly-error branch at 1800-1802 logs and continues writing later lines without advancing currentAddr (so later patches land at the wrong offset), and every `return result` after Phase 3 begins leaves the target half-patched with only a partial disableInfo that the caller must manually replay through disable(). The preprocessing chain (expandLuaBlocks/expandConditionalBlocks/resolveAnonymousLabels) is invoked by check() but NOT by execute() or expandStructDefinitions's callers in execute(), making check() and execute() two materially different front-ends over the same script — the core design smell here. customCommands_/globalSymbols_/knownAllocations_ are shared mutable state with no synchronization, acceptable only under the current single-threaded GUI usage.

### debug

_Files: debug (ptrace debugger, breakpoints, code-finder, tracer, gdb-remote, Intel-PT, LBR, stack-trace)_

The breakpoint/managed-breakpoint/stack-trace pieces are reasonably careful (good overflow guards in stack_trace, snapshot-then-callback pattern in BreakpointManager). The ptrace state machine in DebugSession and the two standalone ptrace loops (CodeFinder, Tracer) are the weak spots: they share a tracee across threads, issue ptrace from the wrong thread, mis-handle waitpid, and have a use-after-free-grade self-detach race. BreakpointManager's lock discipline (copy under lock, run Lua + fire callback outside lock) is the cleanest part; stack_trace's frame-walker validation is solid. Perf ring-buffer drains hand-roll record parsing instead of using a tested helper, and the LBR variant has off-by-design bounds.

### lua-aux

_Files: lua-aux (scripting Lua bindings: Qt GUI bridge, MemoryRecord/AddressList, streams, snapshot, bitmap)_

The five binding files are mostly competent Lua/C glue with correct userdata typing via luaL_checkudata and disciplined stack handling. The MemoryRecord/AddressList layer is notably robust: it stores only a stable integer id and re-resolves the IAddressList from the registry on every call, failing gracefully to nil when detached, so it has no dangling-pointer surface. The bitmap and canvas pixel paths are properly bounds-checked. The serious problems are concentrated in the Qt GUI bridge (lua_gui.cpp): widget/canvas userdata wrap raw QWidget* with no liveness guard, producing a reachable use-after-free once a WA_DeleteOnClose form or a parent widget is destroyed, plus a reentrancy hazard where Qt callbacks call back into the single shared lua_State (e.g. while showModal()/exec() spins a nested event loop). The stream reader also performs an unbounded pre-clamp allocation from an attacker-controlled length.

**Design notes.** Two distinct ownership models are in play. lua_memrec.cpp uses the safe "handle = stable id, look up the owner each call" pattern and tolerates teardown; lua_snapshot uses a shared_ptr holder with placement-new/__gc (correct). lua_streams/lua_bitmap own heap objects (std::vector/std::fstream/QImage) via raw new + __gc delete (correct, single owner). lua_gui.cpp breaks the pattern: LuaWidget/LuaCanvas embed an UNguarded raw QWidget* even though the file already imports QPointer and uses it for guiMainForm and LuaWidgetFont — the guard was simply not applied to the primary widget wrapper, and there is no __gc and no destroyed-time invalidation of the userdata. Callback bookkeeping lives in process-global std::unordered_map<QObject*,...> keyed by raw pointer, and invocation funnels through one shared static guiLuaState (set in registerLuaGuiBindings) with no reentrancy guard. All bindings assume a single Lua thread; the GUI callbacks are dispatched from the Qt event loop, which is the same thread in normal use but becomes reentrant under nested event loops (modal dialogs, exec()).

### analysis

_Files: analysis (code_analysis, managed_runtime, structure_tools)_

This subsystem is memory-safe by construction: every remote-read path correctly clamps work to the actual byte count returned by ProcessHandle::read (the *rr / *read value) rather than the requested size, so there are no partial-read out-of-bounds reads despite process_vm_readv's short-read semantics. Bounds math in structure_tools (compareStructureSnapshots, formatStructureFieldValue, readSnapshotValue, isReadablePointer, readCString) is careful and uses unsigned-safe subtraction guards, and pointer-chain following has proper depth bounds plus a readability check before each dereference. The findings are correctness/completeness and resource issues, not corruption: the strongest is that dissectModule's else-if chain silently discards the GOT/PLT-style indirect call qword ptr [rip+disp] form that is ubiquitous in any PIC shared object, which corrupts the result of the entire code-reference feature set. No critical memory-corruption or RCE-class bug was found.

**Design notes.** Clean separation: code_analysis disassembles executable regions via Capstone and parses operand text strings to recover targets; managed_runtime probes Mono/CoreCLR by string-matching module names then scanning writable regions for pointer-sized values that fall inside runtime image ranges (a heuristic type-handle test); structure_tools handles CT-template I/O, snapshot diffing, C++ struct codegen, and pointer-chain walking. Design strength: all remote reads are validated against returned counts and region bounds; the managed-object scan is properly bounded by maxObjects and an optional heap window. Design smell: target recovery in code_analysis is done by textual parsing of Capstone op_str (parseDirectTarget / parseRipRelativeTarget / parseInteger) instead of consulting Capstone's structured detail (cs_x86 operands / cs_insn detail), which is fragile and is the root cause of finding #1 — CS_OPT_DETAIL is even enabled in the Disassembler but never used here. The detection module duplicates its name-needle lists in two places (isRuntimeModuleName vs detectManagedRuntimes) which can drift. enumerateManagedObjects's whole-region buffer allocation is the only resource concern. The managed type-handle heuristic (any pointer into a runtime image counts as an object header) will produce heavy false positives on real heaps, but that is a heuristic-quality matter, not a safety bug.

- **[MEDIUM] Indirect call/jmp through [rip+disp] (GOT/PLT form) is silently dropped, corrupting all code-reference results** — `analysis/code_analysis.cpp:107-132`  
  dissectModule classifies instructions with an else-if chain. ### cli-arch

_Files: cli-arch (cli/main.cpp, arch/disassembler.{cpp,hpp}, arch/assembler.{cpp,hpp})_

The arch/* Capstone and Keystone wrappers are small and clean: handles are correctly opened/closed, the engine-allocated buffers (cs_free/ks_free) are released on the success path, copies into std::vector are bounds-correct, and apparent type-pun/mode concerns (csh==size_t, CS_MODE_ARM==CS_MODE_LITTLE_ENDIAN==0 for ARM64) are actually safe. The CLI front-end (main.cpp) is the weak spot: it parses operator-supplied argv with atoi/strtoul and feeds the results directly into vector sizing and the scan loop without validation, yielding a reachable infinite-loop hang (--align 0), unbounded-allocation terminates, and a silent no-op/wrong write for several value types. No memory corruption, RCE, or privilege-escalation path was found; severities top out at medium.

**Design notes.** arch/ is a thin RAII wrapper over Capstone/Keystone with non-copyable handles and std::expected error reporting on the assembler — a good, modern shape. handle_ is stored as size_t to keep the C library types out of the header, which is fine because both csh and ks_engine* round-trip losslessly through size_t on LP64. main.cpp is a classic getopt+positional dispatcher; the dispatch guards arg counts (argc >= N) before indexing, so missing-arg OOB is avoided. The systematic weakness is that numeric argv conversion is done with atoi/strtoul/strtoull with no range or error checking, and the converted values flow straight into allocation sizes and loop strides. customValueSize (line 292) is correctly clamped with std::max<size_t>(1, ...); the same discipline is simply not applied to alignment, read size, or disasm count. typeSize() collapses all non-scalar types to 4 with a default case, which is benign for the scan display path but produces a silent wrong write in cmd_write.

### lua-core

_Files: lua-core (scripting/lua_engine.{hpp,cpp}, scripting/lua_bindings.cpp)_

The lua-core subsystem is the C<->Lua boundary exposing CE's memory/scan/debug/process API to scripts. It is generally careful about Lua stack discipline (consistent pop counts, luaL_check* for most typed args, placement-new + __gc for userdata, luaL_ref/unref for callback lifetimes). The dominant systemic weakness is exception safety across the C ABI: Lua is statically linked from C-compiled liblua.a (CMakeLists.txt:122,211), so a C++ exception that escapes a lua_CFunction unwinds through non-exception-aware C frames — undefined behavior. ### platform-core

_Files: platform-core (Linux process/ptrace/injector/kernel-driver/process-watcher)_

The platform-core layer is the security-critical low-level substrate: process_vm_readv/writev memory access, /proc parsing, ptrace debug control, a ptrace+dlopen injector, and a kernel-driver ioctl client. The /proc parsing and the kernel-driver client are reasonably defensive. ### platform-net-overlay

_Files: platform-net-overlay (ceserver client/process/debugger, mono debugger client, network compression, Vulkan overlay)_

### plugins



The plugins subsystem comprises a small dlopen-based .so loader with a clean C ABI (plugin_loader.cpp/.hpp, cecore_plugin.h, sample_plugin.c) plus three LD_PRELOAD interposers (speedhack.c, audiohack.cpp, inprocess_veh.c). The loader and ABI are solid: ABI-version checked, null-checked, std::expected results dereferenced correctly. The interposers carry the real risks: speedhack has an init re-entrancy bug that corrupts its time baseline, lazy dlsym resolution that is not thread-safe and can crash if RTLD_NEXT yields NULL, a hard-coded world-writable /dev/shm region that lets any local user steer or DoS the speed of every hooked process, and a signed-overflow time computation. audiohack mishandles big-endian PCM and has unsynchronized global state. inprocess_veh's signal handler chains to previous handlers and uses fprintf from async-signal context, which is unsafe.

**Design notes.** The structured plugin ABI is well designed: a single host vtable threaded through an opaque host_token, abi_version checks on both sides, NULL-tolerant info fields, and a documented legacy fallback. The loader owns a single heap CePluginHost for its lifetime and dereferences std::expected via operator bool, which is correct. Two structural weaknesses stand out. (1) The interposers reinvent the same lazy `static fn real = NULL; if(!real) real = dlsym(RTLD_NEXT,...)` idiom in ~12 places with no synchronization and inconsistent NULL handling (some return on NULL real_fn, clock_gettime/gettimeofday/nanosleep dereference it unconditionally) — this should be a single GCC constructor-time resolver or use of __attribute__((constructor)) like audiohack/inprocess_veh already do. (2) The /dev/shm/ce_speedhack channel is an unauthenticated, world-writable (0666) cross-process control surface with no per-target namespacing — a trust/security smell for a tool that frequently runs against untrusted game binaries on multi-user hosts. The dlopen path in loadPlugin uses RTLD_NOW|RTLD_LOCAL (good) but performs no path/ownership validation; acceptable only because the load directory is operator-chosen.

### scanner

_Files: scanner (memory_scanner, pointer_scanner, snapshot, cuda_search)_

The scanner subsystem is generally well-constructed: buffer-scan inner loops compute their `limit = bufSize - sizeof(T) + 1` with explicit `bufSize < sizeof(T)` guards, the grouped/AOB/binary paths validate sizes before indexing, and the file-backed ScanResult format keeps address/value/first-value streams aligned. I found no critical memory-corruption bug reachable on realistic input. The real issues are: unchecked `::read()`/`pread()` return values that can silently produce wrong scan results on short reads (EINTR), an unbounded per-region `buf.resize(region.size)` inside worker threads with no exception guard (terminate on bad_alloc), a per-candidate Lua VM creation in the Custom first-scan hot loop (severe perf), and several correctness gaps in the "All types" float filtering and percentage/rounding comparisons. Concurrency is sound by construction — each worker writes its own ScanResult and they are merged single-threaded after join, so there is no shared mutable result state.

**Design notes.** Design strength: the multi-threaded firstScan deliberately gives each thread a private ScanResult writing to its own temp directory, then merges by raw file concatenation after join(). This sidesteps result-aggregation races entirely — the only shared state is the atomic `scannedBytes`/`progress_`/`cancelled_`, all used correctly with relaxed ordering. Buffer reuse is per-thread (local `std::vector<uint8_t> buf`), so no cross-thread buffer aliasing. The on-disk ScanResult triple-stream (addresses/values/first_values) is kept consistent because addResult always appends to all three. Smells: (1) Region-to-thread assignment is by cumulative byte budget, but a single oversized region is never split, so one thread can dominate wall-clock and balloon memory; there is no intra-region chunking. (2) ScanResult opens/closes the backing file on every single address()/value()/firstValue() call — O(open+close) per row, fine for the 10k GUI cap but pathological for large result sets. (3) evaluateCustomFormula spins up a full luaL_newstate()+openlibs() per candidate offset. (4) Loader/parsers (loadPointerPaths, Snapshot::load) are appropriately defensive with magic/version/bound checks — good handling of untrusted serialized input. The CUDA path is a clean stub; the real .cu isn't in scope, but the host-side candidate post-filter re-validates every GPU-reported offset with `off + 8 > bytesRead` and isInRange, so a buggy/malicious kernel cannot cause OOB on the CPU side.

- **[MEDIUM] Unchecked ::read()/pread() return values silently corrupt scan results on short reads** — `scanner/memory_scanner.cpp:1291-1296, 977-978, 935, 944, 955`  
  nextScan() reads the previous result's addresses/values/first_values from temp files with ::read(afd, addrs.data(), n*sizeof(uintptr_t)) etc. and never checks the byte count returned. read() on a regular file can return fewer bytes than requested when interrupted by a signal (EINTR), and pread() in ScanResult::address/value/firstValue is likewise unchecked. A short or interrupted read leaves stale/zeroed bytes in the addr/old-value buffers, so the loop then reads an arbitrary process address (addrs[i] is garbage) and compares against garbage old values — producing wrong results or reading from unintended target addresses, with no error surfaced. forEach() has the same pattern. Within one batch the three streams can also desync (addr read full, value read short), pairing addresses with the wrong values.  
  _Fix:_ Wrap each ::read/pread in a loop that retries on EINTR and accumulates until the requested count is satisfied or EOF/real error; on a confirmed short read (genuine EOF before the expected count) treat the result file as truncated and abort the scan with an error rather than continuing. A small readFull(fd, buf, n) helper returning bool covers all call sites.
- **[MEDIUM] Worker thread allocates buf.resize(region.size) with no bound or exception guard — bad_alloc terminates the process** — `scanner/memory_scanner.cpp:1146-1158, 1036-1046`  
  Each firstScan worker does buf.resize(region.size) then proc.read(region.base, buf.data(), region.size) for the whole clipped region in one shot. A region clipped against the default stopAddress (0x7FFFFFFFFFFF) can be very large, and a single legitimate multi-GB region (common in big games) forces a multi-GB allocation per worker. If resize() throws std::bad_alloc, the exception escapes the std::thread lambda (which has no try/catch), and an exception leaving a thread function calls std::terminate() — the whole process dies mid-scan. The same single-shot full-region read also defeats cancellation responsiveness for huge regions.  
  _Fix:_ Chunk region reads (e.g. 16-64 MB windows with sizeof(T)-1 byte overlap so cross-window matches aren't missed) instead of one buf.resize(region.size); cap the per-read buffer and loop. Wrap the worker body in try/catch and record the error into the per-thread ScanResult / a shared atomic error flag so a failed thread degrades gracefully instead of terminating.
- **[MEDIUM] evaluateCustomFormula creates a fresh Lua state (luaL_newstate + openlibs) for every candidate offset** — `scanner/memory_scanner.cpp:638-665, 1101-1114`  
  In the Custom first-scan path the loop calls evaluateCustomFormula() once per alignment step across the entire region, and each call does luaL_newstate(); luaL_openlibs(); luaL_dostring(formula); lua_close(). Standing up and tearing down a full Lua VM (with all standard libs) for every byte offset is enormous overhead — effectively unusable for any real region — and luaL_dostring recompiles the formula string every time. The same per-call pattern recurs in nextScan's Custom Exact branch.  
  _Fix:_ Create one lua_State per thread (or per scan), compile the formula once with luaL_loadstring into a reusable chunk, and on each candidate set the current/old/hasOld/valueSize globals (or pass them as call args) and lua_pcall the precompiled function. - **[MEDIUM] All-types scan magnitude filter excludes zero and small floats, dropping legitimate matches** — `scanner/memory_scanner.cpp:405-415`  
  scanBufferAllTypes gates every float/double candidate on std::abs(v) > 1e-15 / > 1e-100 in addition to finiteness. This unconditionally discards exact 0.0 and any genuinely small magnitude value before the comparator even runs, so an All-types scan for float 0.0 (a very common search) can never match the float interpretation, and small floats are silently skipped even for Unknown/Changed comparisons where the heuristic 'looks like a plausible float' filter is inappropriate. The filter is applied regardless of compareType.  
  _Fix:_ Only apply the 'plausible float' magnitude heuristic when it is actually wanted (e.g. Unknown/initial scans), and special-case 0.0 so exact searches for zero work. Better: drive the float branch off the requested comparator and let cmpExact handle 0.0 directly; restrict the magnitude squelch to the Unknown case.

### tables-trainer-expr

_Files: tables-trainer-expr (core/ct_file, core/trainer, core/expression, core/types, core/address_list)_

The .CT/.CETRAINER loader, the C-trainer code generator, and the address-expression evaluator. The XML/JSON parsing is hand-rolled and mostly bounds-safe (substr/find with npos guards, no raw pointer indexing), but it treats untrusted CT input as if it were well-formed: at least one numeric conversion is un-try/catched on the live GUI load path, so a hostile table is a one-line denial-of-service. The trainer generator is the most serious problem area: it splices the untrusted CheatEntry `value` string verbatim into generated C source and then invokes gcc, which is an arbitrary-code-execution channel from a crafted .CT. The expression evaluator is currently unreferenced by any caller (latent), but contains unguarded std::stoull throws, unbounded recursion, and a ctype-UB. The "protected" CETRAINER format is XOR/PRNG obfuscation with the password hash stored in cleartext, not real encryption.

**Design notes.** Parsing is dependency-free by design (custom XML tag-scanner + a small recursive-descent JSON parser). The JSON parser is reasonable and the XML helpers (getTag/getTagBlocks/xmlUnescape) avoid raw indexing and use npos-checked substr, so they are largely memory-safe even on garbage input. The main architectural smell is inconsistent error discipline: structure/size/offset conversions are wrapped in try/catch but the entry Address conversion is not, and load()/parse() return bool/optional yet can still escape via exceptions. The trainer generator builds source by string concatenation and shells out to gcc; it correctly shell-quotes file paths and C-escapes descriptions/scripts/game name (cString), but leaves the numeric `value` field unescaped on the assumption it is a valid C integer literal — an assumption an attacker controls. ValueType width is ignored in the generated freeze code (always writes 4 bytes). ExpressionParser only supports +/- (no */ /, no precedence, no div-by-zero possible) and is not yet wired into any caller. types.cpp is empty (header-only); address_list.hpp is a clean abstract interface with no issues.

## Test coverage assessment

Testing is real but narrow: cecore_test (~70 cases, passing) covers pure-logic components against an in-memory FakeProcessHandle, protocol clients against loopback stubs, the Lua binding surface, and live cross-process ptrace against a spawned child; scan_test adds a root-only cross-process scan-and-write-back integration check. The sharpest coverage finding is that the confirmed critical bugs sit precisely in the untested tier: the kernel-module ioctls (process-VM, physical-memory, VA-translate, /proc hiding) are entirely unexercised, and the hand-rolled ELF symbol parser — the source of the confirmed heap overflow — is never fed a malformed input. More broadly there is no negative/fuzz testing of ANY parser even where the happy path is covered (ELF, CT XML/JSON, GDB-RSP, the standalone expression evaluator, the Mono soft-debugger protocol). Also untested: the raw-ptrace injector, the CodeFinder/Tracer watchpoint loops, Intel-PT and LBR ring-buffer drain logic (where another confirmed OOB read lives), the entire Qt GUI layer, the Lua snapshot/GUI bindings, and ProcessWatcher. The untested surface and the worst bugs coincide.

## Prioritized recommendations

1. Fix the two confirmed criticals immediately: in elf_symbols.cpp validate sh_entsize/sh_size against file length and require read-size == buffer-size; in cecore_kmod.c make the /proc-hook lifetime safe (RCU/refcount or restore-and-synchronize before unload) and fix the dentry leak.
2. Add pfn_valid/page_is_ram/range and overflow validation to the kernel physical-memory ioctls before any ioremap, and bound all address+offset arithmetic against wraparound.
3. Close the two prose-only RCE channels: stop splicing untrusted CheatEntry strings into gcc-compiled C in the trainer generator, and sandbox/gate the Lua *Local read-write family and shellExecute behind explicit opt-in with a trust warning.
4. Fix the confirmed high-severity memory-safety bugs: guard lua_gui widget pointers with QPointer + null-checks, and correct the LBR drain bound with proper ring-wrap handling.
5. Repair autoasm: run the same preprocessors in execute() as in check(), and gate every patch on a verified full read so original bytes are never zero-filled and restore can never corrupt the target.
6. Apply the untrusted-size rule everywhere: validate every wire/script/argv/region length against a sane cap and the bytes actually available before sizing allocations or reads (ceserver_client, mono_debugger_client, scanner region buffers, lua_streams, cli/main, memoryfill, managed_runtime), and wrap residual throwing allocations.
7. Harden the ptrace layer: consolidate the duplicated logic, check every ptrace return and waitpid status, confirm stops via RIP/DR6, move to PTRACE_SEIZE/INTERRUPT, attach all threads, and stop invoking callbacks under bpMutex_.
8. Sandbox CT-sourced Lua (breakpoint conditions, custom scan formulas) into a restricted environment without os/io/package/loadfile, and reuse a single Lua state in the scanner hot loop.
9. Add an exception firewall around every lua_CFunction (try/catch to lua_error) and null-guard lua_tostring on error objects; implement GDB-RSP RLE/escape decoding with bounded reads; add INCLUDE depth/cycle limits.
10. Lock down root-side file handling: O_NOFOLLOW|O_EXCL, 0600 modes, per-user private directories instead of world-writable /dev/shm, and mkstemp for temp files (mainwindow speedhack, plugins speedhack, ct_file protected temp).
11. Raise the build/defensive baseline: enable -Wall -Wextra -Werror, _FORTIFY_SOURCE, stack-protector, and CI ASan/UBSan plus fuzz harnesses for every untrusted parser; pin third-party dependencies by commit hash rather than mutable git tags.
12. Backfill tests where the worst bugs live: negative/fuzz tests for the ELF, CT, GDB-RSP, expression, and Mono parsers; integration tests for the kernel-module ioctls; and coverage of the injector, watchpoint loops, and PT/LBR drain logic.

## Appendix — refuted findings (2)

Reported by an auditor but **refuted** by the adversarial verifier against the source. Kept for transparency.

- **Reentrancy on the shared lua_State from Qt callbacks (modal dialogs / nested event loops)** (`scripting/lua_gui.cpp:126-140, 277-285, 459-472`, lua-aux) — claimed high.  
  _Why refuted:_ (a) Cited code is accurate. lua_gui.cpp:126-140 is invokeLuaCallback doing lua_rawgeti + lua_pcall(guiLuaState,...). Lines 277-285 are the showModal closure: `if (auto* dlg = qobject_cast<QDialog*>(w)) dlg->exec(); else w->show();` — exec() does spin a nested Qt event loop. Lines 459-472 are the OnTimer handler connecting QTimer::timeout to a lambda that calls invokeLuaCallback. registerLuaGuiBindings (line 734) sets guiLuaState = L, and callers pass engine->state()/luaEngine_.state() (gui/luaconsole.cpp:38, gui/mainwindow.cpp:707), so guiLuaState IS the same lua_State L_ that execute()/executeFile() run luaL_dostring/luaL_dofile on (lua_engine.cpp:254/292). LuaConsole::onExecute (luaconsole.cpp:41-51) is a Qt slot, so the whole thing runs on the GUI thread inside the event loop. So the described reentry PATH is real and reachable: a script calling showModal() while inside luaL_dostring, with an OnTimer/OnClick firing during the nested exec() loop, will call lua_pcall on a lua_State that is already mid-pcall.  (b)/(c) BUT the central technical claim is wrong. This is purely SYNCHRONOUS, SINGLE-THREADED nested reentry on the C stack — not concurrency, not async/signal interruption. There are no threads (grep found no std::thread/QThread/moveToThread/QtConcurrent in scripting/ or the engine; no lua_newthread). When the script calls showModal(), control has already returned from the Lua interpreter loop into the C function via the normal call protocol; the timer callback's lua_pcall is a fresh, properly stack-balanced nested call. Nested C->Lua->C->Lua lua_pcall calls on the SAME lua_State from the SAME thread are a fully supported, designed-for part of the Lua 5.3 C API (each lua_pcall saves/restores the stack and links CallInfo). It is NOT undefined behaviour and does NOT corrupt the interpreter stack. Lua UB requires either concurrent access from another OS thread (absent here) or true asynchronous interruption mid-instruction such as a signal handler (also absent — event-loop reentry only happens at well-defined points where the C-API state is consistent). So 'undefined behaviour / corrupts the interpreter stack / can crash' and the 'concurrency' category are both incorrect/inflated.  A genuine but MILD residual concern remains: reentrant application-level logic — a callback executing while a script is still mid-run can confuse app invariants. That is a low-severity quality issue, not the high-severity interpreter-corruption bug claimed. (Use-after-free is not a factor: trackDestroyed + Qt's automatic sender-bound disconnect prevent a destroyed widget from firing its own callback.)  (d) Tests don't exercise GUI/Lua-GUI, so they neither confirm nor contradict; but the finding's load-bearing premise is technically false regardless. Refuted because the specific claimed mechanism (UB/stack corruption/crash from reentering the lua_State) does not hold; at most a low quality concern.
